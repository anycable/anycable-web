Should you switch? Stay on Action Cable or Solid Cable if you serve well under ~200K connections, deploy off-hours, and an occasional dropped update is acceptable. Reach for AnyCable when any of these holds: you deploy during business hours (every in-process deploy drops 100% of connections at once), your users are on flaky mobile or WiFi and silent message loss is unacceptable (chat, Turbo Streams, AI token streaming), or you are approaching ~200K connections. How it was measured
Each adapter is the same Rails 8.1 app on its own Railway container (32 GB plan limit). The load drivers are separate Railway services hitting it over the private network in the same region, so latency reflects the server and not the public internet. Puma runs 8 workers × 5 threads; Falcon runs 8 processes; AnyCable pairs anycable-go with a Rails gRPC backend. The in-process adapters speak actioncable-v1-json (via actioncable-next for the Falcon target); AnyCable speaks actioncable-v1-ext-json. Each adapter is driven by the JS client a real app of that kind ships: the official @rails/actioncable for Action Cable, Solid Cable and Async::Cable, and @anycable/core for AnyCable. That choice only moves the reconnect-driven tests (jitter and deploy); latency and capacity are client-independent. Latency and jitter cells are ~100 messages per client at 500 ms intervals, one sharded pass per cell within a single shared-tenant window (point estimates, not multi-run averages). Capacity-to-break ramps idle connections until the box fails, scaling the driver count to the target (~1 driver per 10K connections). What we compare#

Four adapters for the same Rails 8.1 + Hotwire app, same hardware. The channel code, the stream_from calls, and the Turbo Stream broadcasts are identical in all four. What changes is one line in config/cable.yml, and what process holds the sockets:

  • Action Cable on the Redis adapter · the Rails default. WebSockets terminate in-process, inside Puma’s Ruby threads; Redis carries pub/sub.
  • Solid Cable · the database-backed adapter shipped with Rails 8. Same in-process termination, no Redis; the adapter polls a database table (default every 100 ms).
  • Async::Cable · async-cable serves Action Cable in-process on Falcon, a fiber-based reactor, instead of Puma’s threads. Same wire protocol; a different concurrency engine.
  • AnyCable in full RPC mode · Rails runs a gRPC backend while anycable-go, a separate Go process, holds the WebSockets and fans out broadcasts.
All four are Action Cable-compatible at the app and client level: same channels, same JS client, Turbo Streams unchanged. On the wire they differ. Action Cable, Solid Cable and Async::Cable speak the base actioncable-v1-json protocol; AnyCable speaks the extended actioncable-v1-ext-json protocol, which adds resume and delivery guarantees. That one difference drives the reliability gap below.

{{!-- Topology + wire-protocol diagram. --}}

It comes down to two seams: where the socket lives, and what the wire remembers

Topology. Action Cable, Solid Cable and Async::Cable terminate WebSockets inside the same process that serves your HTTP requests, whether that process runs Puma threads or Falcon fibers. When you deploy, that process restarts and every socket drops. AnyCable moves the sockets to a separate Go gateway; Rails only handles auth and commands over gRPC, so an app deploy never touches a live connection.

Wire protocol. The base actioncable-v1-json protocol is at-most-once: a message broadcast while a client is briefly offline is gone. The extended actioncable-v1-ext-json protocol AnyCable speaks keeps per-stream history with an epoch and offset, so a reconnecting client resumes from the last message it saw. The JS client negotiates the protocol automatically.

01 · How fast is it?#

Latency is the floor of what realtime can feel like. We measured roundtrip time from a publisher broadcasting through Rails to the moment each subscriber received the message, on a clean network, at 1K and 5K subscribers.

AnyCable is the fastest at both sizes. Fan-out happens in the Go gateway rather than in Ruby, so it stays at single-digit milliseconds where the in-process adapters climb with load. Action Cable on Puma is close behind; Async::Cable on Falcon sits a touch higher; Solid Cable carries a fixed floor from its 100 ms database poll. Measured with a sharded driver fleet so the number reflects the server itself. Latency is where the four sit closest: the AnyCable-to-Action-Cable gap (7 vs 13 ms) is small enough that few users would feel it. The separation that decides production shows up under jitter and on deploys, below.

02 · How reliable is delivery?#

Your users are on hotel WiFi, the subway, a phone with two bars. Those micro-drops are invisible to a page load and they are the whole game for realtime. The question is what happens to a Turbo Stream broadcast that goes out while a client is offline for a couple of seconds.

We held 5,000 subscribers and dropped each one’s connection for about 2 seconds every ~15 seconds while broadcasting steadily. This is a deliberate stress test: every client flapping on a tight cycle, to isolate how each protocol behaves under loss, not a steady-state average. Each adapter runs the JS client a real app of that kind ships, which matters here: the official @rails/actioncable client reconnects on its own poll-based schedule, so the client stays offline a few seconds past the 2 second drop, and the base actioncable-v1-json protocol has no resume, so whatever landed during that window is gone. All three in-process adapters behave identically, because the loss lives in the base protocol and the shared default client; the Puma-or-Falcon runtime makes no difference. AnyCable pairs its faster @anycable/core client with the extended protocol, which replays each stream’s history from the last offset on reconnect, so nothing is dropped.

AnyCable delivers 100% under jitter. The three at-most-once adapters land at the same ~53% with their real default client, losing the broadcasts that arrive during each offline window. Swapping Puma for Falcon leaves that unchanged; only the resume protocol moves it.

Action Cable Puma 53% 12 ms n/a no resume

The in-process p50 is fast on the messages that land; the ~47% that arrive during an offline window, stretched wider by the stock client’s slower reconnect, are simply gone. AnyCable’s p99 is the resume tail: history replayed on reconnect arrives about two seconds late, completing the stream rather than dropping it. Methodology.

{{!-- What this breaks --}}

What this breaks

Half of broadcasts lost sounds abstract until you see where it lands. Loss clusters around network events, exactly when a user is looking at the screen. The base protocol has no resume, so the standard fix is to refetch the whole stream on reconnect; resume just replays the few messages that were missed, automatically and incrementally. Here is what you build around otherwise.

Hotwire Turbo Streams A broadcast_append_to during a blip never lands, so the list misses a row until the page refetches. Resume replays the missed frame on reconnect, no full reload.
Live chat & notifications A message sent during the blip won’t arrive until the client refetches the conversation. Resume delivers it on reconnect without a reload.
LLM / AI streaming Token chunks streamed over a channel arrive truncated after a reconnect. Resume from the last offset is what keeps the response whole.
Deploys at peak hours Every in-process deploy drops all sockets at once. Users mid-action lose live updates for several seconds while the reconnect storm clears.

03 · Does it survive a deploy?#

Rails apps deploy often. Every time an in-process WebSocket server restarts, all of its connections drop at once and reconnect in a thundering herd. We connected 5,000 clients, ran a real railway redeploy on the app mid-test, and measured what happened. For AnyCable the redeploy hit the Rails RPC backend while the gateway kept holding the sockets. Background on the failure mode in Vladimir’s post at Martian Chronicles.

Every in-process adapter drops all 5,000 connections on deploy; what matters is the recovery. Getting 95% back takes ~13.5 to 14 seconds with the stock Action Cable client, whose poll-based reconnect is the slow part, and about 4% never reconnect within the window at all. Falcon behaves like Puma here: swapping the runtime leaves the deploy fragility intact. AnyCable’s gateway never sees the deploy, so there is no outage and nothing to reconnect.

Adapter 5K clients, app redeploy Recovery to 95% reconnect Reconnected within the window
Action Cable Puma 13.5 s 96.6% 172 still out
Solid Cable DB 13.8 s 96.5% 177 still out
Async::Cable Falcon 13.8 s 96.1% 193 still out
AnyCable 0 s n/a never dropped

5,000 clients connected, then a real Railway redeploy of the app service. Every in-process adapter dropped all 5,000 connections; “Recovery” is the wall-clock from the redeploy until 95% of clients had reconnected on the stock @rails/actioncable client, whose poll-based reconnect is the slow part, and “Reconnected” is the share back online when observation stopped (the “still out” clients never came back within the window). The reconnect storm is generated across 20 drivers, so the recovery time is the adapter’s, not the load fleet’s. AnyCable was redeployed the same way, on its Rails RPC backend and driven by @anycable/core: the gateway reported no disconnect at all, so there was zero downtime and nothing to reconnect.

04 · How far does it scale?#

In-process Action Cable has a reputation for falling over under load. With a properly sharded load generator it does not at the sizes most apps see: all four adapters held 10,000 subscribers at 100% delivery while broadcasting, and AnyCable kept the tightest tail. At that scale, reliability and deploys are what separate them.

Pushed to failure on identical 32 GB boxes, the picture changes. The two Puma adapters wall around ~210K connections on an 8-worker file-descriptor ceiling, reached before the memory limit. Async::Cable on Falcon walls first, at ~105K, because each fiber-backed connection costs roughly 290 KB, six times the others, so on raw capacity Falcon trails the Puma pair. AnyCable held 792K with zero failures before we ran out of load generators, using the whole box (one Go process, no per-worker wall) at about 47 KB per connection. Same per-connection cost as Puma, several times the headroom.

FAQ#

Which Rails WebSocket adapter should I choose?
For most Rails apps the choice is about reliability and operations, not raw speed, since all four hold 10,000 subscribers at 100% delivery. Choose Solid Cable if you want zero extra infrastructure (no Redis, just your database) and can accept message loss on reconnect and on deploy. Choose Action Cable on Redis or Async::Cable on Falcon if you are already invested in that runtime; both still lose about half of messages under network jitter with their default client and drop every connection on deploy. Choose AnyCable when you need reliable delivery (100% under jitter), connections that survive deploys, built-in presence, or scale past ~200,000 connections; it is the same Action Cable API with an out-of-process Go gateway. AnyCable is the safest default for production realtime on Rails.
What is AnyCable, and how is it different from Action Cable?
AnyCable is an open-source WebSocket server for Ruby and Rails apps, built by Evil Martians and MIT-licensed since 2017. It speaks the Action Cable API, so your channels, stream_from calls, and Turbo Streams are unchanged, but it runs the WebSocket connections in a separate Go process (anycable-go) instead of inside Puma. Compared with Rails’ built-in Action Cable it adds reliable message delivery with resume, connections that survive app deploys, built-in presence, and far higher connection capacity (792,000 in this benchmark versus about 217,000 for in-process Action Cable).
Is AnyCable a drop-in replacement for Action Cable?
At the app and client level, yes. Your channels, stream_from calls, and Turbo Streams stay exactly as written. You add the anycable-rails gem, set adapter: any_cable in config/cable.yml, run the Rails RPC backend with bundle exec anycable, and run anycable-go as the WebSocket server. No channel rewrite. Rollback is the same one-line revert: a stock Action Cable client reconnects over the base protocol.
Do these adapters speak the same wire protocol?
They are Action Cable-compatible at the app and client level, so the same JS client works with all four. On the wire they differ. Action Cable, Solid Cable and Async::Cable speak the base actioncable-v1-json protocol, which is at-most-once with no resume. AnyCable speaks the extended actioncable-v1-ext-json protocol, which adds per-stream history, an epoch and offset, and resume on reconnect. The AnyCable JS client negotiates the extended protocol automatically; a plain Action Cable client still connects over the base protocol.
What is Async::Cable, and does Falcon fix Action Cable’s problems?
async-cable serves Action Cable on Falcon, a fiber-based Ruby server, instead of Puma’s threads. It is a real alternative runtime, and in our tests its latency sits in the same range as Puma Action Cable, though its raw capacity is a bit lower since fibers cost more memory per connection. It changes only the runtime, leaving the guarantees untouched: it is still in-process (so it drops every connection on deploy) and still speaks the base protocol (so it loses about half of messages under jitter, the same ~53% as the others). Falcon does not close the gap to AnyCable on reliability or deploy survival, because those gaps come from the protocol and the topology.
Doesn’t in-process Action Cable fall over at scale?
At everyday scale, less than its reputation suggests: all four adapters held 10,000 subscribers at 100% delivery with zero failures, so for most apps capacity rarely decides it. Pushed to the limit on identical 32 GB boxes, the difference is large. Puma Action Cable and Solid Cable wall around 210K connections on an 8-worker file-descriptor ceiling, reached before the memory limit. Async::Cable on Falcon runs out of memory first, near 105K (its fibers cost ~290 KB each). AnyCable held 792K with zero failures before our load fleet maxed out, using the whole box at ~47 KB per connection. So in-process Rails is fine until you need real scale; past ~200K, AnyCable’s separate Go process is what carries it, on top of its latency, jitter, and deploy wins.
How many concurrent connections can Action Cable handle?
In this benchmark, in-process Action Cable on Puma held about 217,000 concurrent connections on a 32 GB box running its full 8 workers, hitting a file-descriptor ceiling before the memory limit (raising the fd limit lifts it further). Solid Cable on Puma reached a similar ~206K; Async::Cable on Falcon reached ~105K before running out of memory, the lowest of the in-process trio because each fiber-backed connection costs roughly six times the RAM. AnyCable, which holds connections in a separate Go process, reached 792,000 with zero failures and was still climbing when the load fleet maxed out.
Do I have to write or run Go to use AnyCable?
No. You write Ruby; anycable-go ships as a prebuilt binary and Docker image that you run as a service, the same way you run Redis or Postgres. You never touch Go code. The only operational addition over Action Cable is running that gateway process (and a broker like Redis or NATS if you scale to multiple gateway instances).
Does the 100% delivery require changing my JavaScript client?
The resume guarantee rides on the extended actioncable-v1-ext-json protocol, which the AnyCable JS client negotiates automatically. It is a drop-in replacement for @rails/actioncable, and the AnyCable Turbo Streams package wires it into Hotwire for you, so it is a dependency swap, not a rewrite. A stock @rails/actioncable client still connects and works, but over the base protocol without resume, so for the 100% figure you do ship the AnyCable client.
Is Solid Cable slower than Action Cable?
A little, and predictably so. Its database-polling adapter adds a fixed floor set by the poll interval (100 ms by default), measuring 88 ms p50 at 5K subscribers versus 11 ms for Action Cable on Redis. In return you run no Redis at all, just your existing database. For low-to-moderate broadcast volume that is a reasonable trade. It shares the in-process limits: no resume, and connections drop on every deploy.
Do my connections really survive a deploy with AnyCable?
Yes. Because the WebSockets live in anycable-go and only auth and commands go to Rails over gRPC, deploying your Rails app restarts the RPC backend without touching the gateway. In our test, redeploying the Rails app dropped zero of 5,000 connections; the same redeploy on the three in-process adapters dropped all 5,000 and took about 13.5 s to reconnect ~96% on the stock Action Cable client, leaving roughly 180 clients stranded.

Does AnyCable work with Hotwire and Turbo Streams?
Yes, with no changes. turbo-rails broadcasts through Action Cable’s API, and AnyCable implements that API, so broadcast_append_to, broadcast_replace_to, and the rest route through the gateway transparently. The upside is that those broadcasts now resume across reconnects, which is exactly where vanilla Turbo Streams silently lose updates.
How much does this cost versus Pusher or Ably?
Pusher and Ably charge per concurrent connection, so cost scales linearly with your user base. AnyCable Pro is a flat-rate self-hosted license ($1,490/yr, with unlimited connections and instances) and a 2-month free trial; AnyCable+ Managed is free for early users. How do I run anycable-go with Rails in production?
Run two processes: your Rails app with bundle exec anycable as the gRPC RPC backend, and anycable-go as the WebSocket server pointed at it via ANYCABLE_RPC_HOST. Both ship as Docker images. Set a shared ANYCABLE_SECRET for auth, add Redis or NATS as the broker when you run multiple gateway instances, and put the gateway behind a load balancer. Health endpoint at /health, Prometheus metrics at /metrics, graceful drain on SIGTERM. The Rails guide has Railway, Fly and Kubernetes templates.
Is AnyCable open source?
Yes, MIT-licensed since 2017. anycable/anycable and anycable/anycable-rails on GitHub. A commercial Pro tier adds broker features (long-term history, embedded NATS) and priority support, free for small deployments.
