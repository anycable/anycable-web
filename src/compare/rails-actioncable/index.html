What we compare#

Four adapters for the same Rails 8.1 + Hotwire app, same hardware. The channel code, the stream_from calls, and the Turbo Stream broadcasts are identical in all four. What changes is one line in config/cable.yml, and what process holds the sockets:

  • Action Cable on the Redis adapter · the Rails default. WebSockets terminate in-process, inside Puma’s Ruby threads; Redis carries pub/sub.
  • Solid Cable · the database-backed adapter shipped with Rails 8. Same in-process termination, no Redis; the adapter polls a database table (default every 100 ms).
  • AsyncCable · async-cable serves Action Cable in-process on Falcon, a fiber-based reactor, instead of Puma’s threads. Same wire protocol; a different concurrency engine.
  • AnyCable in full RPC mode · Rails runs a gRPC backend while anycable-go, a separate Go process, holds the WebSockets and fans out broadcasts.
All four are Action Cable-compatible at the app and client level: same channels, same JS client, Turbo Streams unchanged. On the wire they differ. Action Cable, Solid Cable and AsyncCable speak the base actioncable-v1-json protocol; AnyCable speaks the extended actioncable-v1-ext-json protocol, which adds resume and delivery guarantees. That one difference drives the reliability gap below. 
# config/cable.yml
production:
  adapter: redis
  url: <%= ENV["REDIS_URL"] %>

# Broadcast from anywhere, unchanged:
Turbo::StreamsChannel.broadcast_append_to(
  "room:42", target: "messages",
  partial: "messages/message"
)

# threads. Wire: actioncable-v1-json, no resume.
# config/cable.yml, no Redis

Two things differ: topology and wire protocol

Topology. Action Cable, Solid Cable and AsyncCable terminate WebSockets inside the same process that serves your HTTP requests, whether that process runs Puma threads or Falcon fibers. When you deploy, that process restarts and every socket drops. AnyCable moves the sockets to a separate Go gateway; Rails only handles auth and commands over gRPC, so an app deploy never touches a live connection.

Wire protocol. The base actioncable-v1-json protocol is at-most-once: a message broadcast while a client is briefly offline is gone. The extended actioncable-v1-ext-json protocol AnyCable speaks keeps per-stream history with an epoch and offset, so a reconnecting client resumes from the last message it saw. The JS client negotiates the protocol automatically.

01 · How fast is it?#

Latency is the floor of what realtime can feel like. We measured roundtrip time from a publisher broadcasting through Rails to the moment each subscriber received the message, on a clean network, at 1K and 5K subscribers.

AnyCable is the fastest at both sizes. Fan-out happens in the Go gateway rather than in Ruby, so it stays at single-digit milliseconds where the in-process adapters climb with load. Action Cable on Puma is close behind; AsyncCable on Falcon sits a touch higher; Solid Cable carries a fixed floor from its 100 ms database poll. Measured with a sharded driver fleet so the number reflects the server itself.

AnyCable leads at every scale (7 ms p50 at 5K) and holds the tightest tail. The in-process adapters stay fast too; Solid Cable’s floor is its database poll.

{{!-- 02 — Reliability under jitter. --}}

02 · How reliable is delivery?#

Your users are on hotel WiFi, the subway, a phone with two bars. Those micro-drops are invisible to a page load and they are the whole game for realtime. The question is what happens to a Turbo Stream broadcast that goes out while a client is offline for a couple of seconds.

We held 5,000 subscribers and dropped each one’s connection for about 2 seconds every ~15 seconds while broadcasting steadily. The base actioncable-v1-json protocol has no resume: whatever landed during an offline window is lost. All three in-process adapters behave identically here, because the loss lives in the base protocol; the Puma-or-Falcon runtime makes no difference. AnyCable’s extended protocol replays each stream’s history from the last offset on reconnect, so nothing is dropped.

AnyCable delivers 99.9% under jitter. The three at-most-once adapters land at the same ~78%, losing the broadcasts that arrive during each offline window. Swapping Puma for Falcon leaves that at 78%; only the resume protocol moves it.

Adapter 5K subs Delivery p50 p99 tail
Solid Cable DB 78.1% 71 ms no resume
Action Cable Puma 78.1% 12 ms no resume
AsyncCable Falcon 78.1% 18 ms no resume
AnyCable 99.9% 7 ms ~6 s

5,000 clients, each dropped for ~2 s every ~15 s while a publisher broadcasts. Delivery is the share of expected broadcasts each client received. The in-process p50 is fast on the messages that land; the ~22% that arrive during an offline window are simply gone. AnyCable’s p99 is the resume tail: history replayed on reconnect arrives a few seconds late, completing the stream rather than dropping it. Methodology.

{{!-- What this breaks --}}

What this breaks

A fifth of broadcasts lost sounds abstract until you see where it lands. Loss clusters around network events, exactly when a user is looking at the screen, and with the base protocol it is silent: no error, no retry, just a Turbo frame that never appends and a UI that quietly falls out of sync.

Hotwire Turbo Streams A broadcast_append_to during a blip never lands. The list silently misses a row; the user sees a stale page with no sign anything is gone.
Live chat & notifications Messages sent while a recipient is offline for a second vanish. Conversations look complete but skip a beat nobody can recover.
LLM / AI streaming Token chunks streamed over a channel arrive truncated after a reconnect. Resume from the last offset is what keeps the response whole.
Deploys at peak hours Every in-process deploy drops all sockets at once. Users mid-action lose live updates for several seconds while the reconnect storm clears.

03 · Does it survive a deploy?#

Rails apps deploy often. Every time an in-process WebSocket server restarts, all of its connections drop at once and reconnect in a thundering herd. We connected 5,000 clients, ran a real railway redeploy on the app mid-test, and measured what happened. For AnyCable the redeploy hit the Rails RPC backend while the gateway kept holding the sockets. Background on the failure mode in Vladimir’s post at Martian Chronicles.

In-process adapters drop all 5,000 connections on deploy and take ~8 s to reconnect about 96%, leaving ~200 clients stranded. Falcon behaves like Puma here: swapping the runtime leaves the deploy fragility intact. AnyCable’s gateway never sees the deploy.

Adapter 5K clients, app redeploy Dropped Reconnected Recovery
Action Cable Puma all 5,000 96.3% 187 lost 7.5 s
Solid Cable DB all 5,000 95.7% 215 lost 7.6 s
AsyncCable Falcon all 5,000 96.4% 179 lost 8.0 s
AnyCable 0 n/a 0 s

5,000 clients connected, then a real Railway redeploy of the app service. “Recovery” is wall-clock until the reconnect storm settles; “lost” clients never reconnected within the observation window. AnyCable was redeployed the same way, on its Rails RPC backend: the gateway reported no disconnect at all, so reconnect time is zero by construction.

{{!-- 04 — Capacity --}}

04 · How far does it scale?#

In-process Action Cable has a reputation for falling over under load. With a properly sharded load generator it does not at the sizes most apps see: all four adapters held 10,000 subscribers at 100% delivery while broadcasting, and AnyCable kept the tightest tail. At that scale, reliability and deploys are what separate them.

Pushed to failure on identical 32 GB boxes, the picture changes. The two Puma adapters wall at ~52K connections while using only ~2.5 GB of RAM: an 8-worker file-descriptor ceiling reached well below the memory limit. AsyncCable on Falcon runs out of memory at ~97K, because each fiber-backed connection costs roughly 290 KB, six times the others. AnyCable held 600K with zero failures before we ran out of load generators, using the whole box (one Go process, no per-worker wall) at about 47 KB per connection. Same per-connection cost as Puma; eleven times the headroom.

All four hold 10K at 100% delivery. Pushed to failure, the Puma adapters wall at ~52K on a file-descriptor limit, Falcon runs out of memory at ~97K, and AnyCable held 600K with zero failures, still climbing when the load fleet maxed out.

FAQ#

Is AnyCable a drop-in replacement for Action Cable?
At the app and client level, yes. Your channels, stream_from calls, and Turbo Streams stay exactly as written. You add the anycable-rails gem, set adapter: any_cable in config/cable.yml, run the Rails RPC backend with bundle exec anycable, and run anycable-go as the WebSocket server. No channel rewrite.
Do these adapters speak the same wire protocol?
They are Action Cable-compatible at the app and client level, so the same JS client works with all four. On the wire they differ. Action Cable, Solid Cable and AsyncCable speak the base actioncable-v1-json protocol, which is at-most-once with no resume. AnyCable speaks the extended actioncable-v1-ext-json protocol, which adds per-stream history, an epoch and offset, and resume on reconnect. The AnyCable JS client negotiates the extended protocol automatically; a plain Action Cable client still connects over the base protocol.
What is AsyncCable, and does Falcon fix Action Cable’s problems?
async-cable serves Action Cable on Falcon, a fiber-based Ruby server, instead of Puma’s threads. It is a real alternative runtime, and in our tests its latency and capacity are in the same range as Puma Action Cable. But it changes only the runtime, leaving the guarantees untouched: it is still in-process (so it drops every connection on deploy) and still speaks the base protocol (so it loses about a fifth of messages under jitter, the same 78% as the others). Falcon does not close the gap to AnyCable on reliability or deploy survival, because those gaps come from the protocol and the topology.
Doesn’t in-process Action Cable fall over at scale?
At everyday scale, less than its reputation suggests: all four adapters held 10,000 subscribers at 100% delivery with zero failures, so for most apps capacity rarely decides it. Pushed to the limit on identical 32 GB boxes, the difference is large. Puma Action Cable and Solid Cable wall at about 52K connections, an 8-worker file-descriptor ceiling reached at only ~2.5 GB of RAM. AsyncCable on Falcon runs out of memory near 97K (its fibers cost ~290 KB each). AnyCable held 600K with zero failures before our load fleet maxed out, using the whole box at ~47 KB per connection. So in-process Rails is fine until you need real scale; past ~50K, AnyCable’s separate Go process is what carries it, on top of its latency, jitter, and deploy wins.
Is Solid Cable slower than Action Cable?
A little, and predictably so. Its database-polling adapter adds a fixed floor set by the poll interval (100 ms by default), measuring 74 ms p50 at 5K subscribers versus 13 ms for Action Cable on Redis. In return you run no Redis at all, just your existing database. For low-to-moderate broadcast volume that is a reasonable trade. It shares the in-process limits: no resume, and connections drop on every deploy.
Do my connections really survive a deploy with AnyCable?
Yes. Because the WebSockets live in anycable-go and only auth and commands go to Rails over gRPC, deploying your Rails app restarts the RPC backend without touching the gateway. In our test, redeploying the Rails app dropped zero of 5,000 connections; the same redeploy on the three in-process adapters dropped all 5,000 and took about 8 s to reconnect ~96%, leaving roughly 200 clients stranded.
How does AnyCable compare on performance?
Four workloads on the same Railway box, every adapter as the same Rails 8.1 app, load sharded across 13 drivers. Roundtrip latency: AnyCable is fastest at every scale (4 ms p50 at 1K, 7 ms at 5K), ahead of Action Cable (9 and 13 ms), AsyncCable on Falcon (11 and 20 ms) and Solid Cable (62 and 74 ms). Jitter at 5K (~2 s drops): AnyCable 99.9% delivered; the three in-process adapters all ~78%. Capacity: all four hold 10K at 100%; pushed to failure on 32 GB boxes, Puma Action Cable and Solid Cable wall at ~52K (file-descriptor limit), Falcon AsyncCable at ~97K (memory), and AnyCable held 600K with zero failures. Deploy survival at 5K: AnyCable 0 dropped, the in-process adapters drop all 5,000 and reconnect ~96% in ~8 s. Reproduce from github.com/anycable/nodejs-websocket-bench.
Does AnyCable work with Hotwire and Turbo Streams?
Yes, with no changes. turbo-rails broadcasts through Action Cable’s API, and AnyCable implements that API, so broadcast_append_to, broadcast_replace_to, and the rest route through the gateway transparently. The upside is that those broadcasts now resume across reconnects, which is exactly where vanilla Turbo Streams silently lose updates.
How much does this cost versus Pusher or Ably?
Pusher and Ably charge per concurrent connection, so cost scales linearly with your user base. AnyCable Pro is a flat-rate self-hosted license ($1,490/yr, with unlimited connections and instances) and a 2-month free trial; AnyCable+ Managed is free for early users. At a few thousand concurrent connections and up, flat-rate self-hosting typically saves thousands per month against per-connection pricing.
How do I run anycable-go with Rails in production?
Run two processes: your Rails app with bundle exec anycable as the gRPC RPC backend, and anycable-go as the WebSocket server pointed at it via ANYCABLE_RPC_HOST. Both ship as Docker images. Set a shared ANYCABLE_SECRET for auth, add Redis or NATS as the broker when you run multiple gateway instances, and put the gateway behind a load balancer. Health endpoint at /health, Prometheus metrics at /metrics, graceful drain on SIGTERM. The Rails guide has Railway, Fly and Kubernetes templates.
Is AnyCable open source?
Yes, MIT-licensed since 2017. anycable/anycable and anycable/anycable-rails on GitHub. A commercial Pro tier adds broker features (long-term history, embedded NATS) and priority support, free for small deployments.
